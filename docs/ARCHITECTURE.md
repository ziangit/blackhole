# event-horizon — architecture (living doc)

> Update this whenever a non-obvious decision or constraint is discovered.

## Stack
- Chrome extension, Manifest V3.
- Plain TypeScript bundled with esbuild (no framework, no bundler config beyond a build script).
- `src/background.ts` — MV3 service worker: mass model, heartbeat accumulation, alarms decay, badge.
- `src/content.ts` + helpers — injected on `https://x.com/*` and `https://twitter.com/*` at `document_idle`: timeline acquisition, hole overlay canvas, lens renderer, heartbeats.
- `src/options.html/ts` — settings page, live-apply via `storage.onChanged`.
- State: `chrome.storage.local` is the single source of truth for `mass` and settings.

## Known constraints & gotchas
- **CSP findings (Milestone 0):**
  - x.com's response CSP (`curl -I https://x.com/`, 2026-06-11) has `img-src 'self' blob: data: …` — **`data:` and `blob:` URLs are explicitly allowed**, so strategy 1 (feImage + canvas `data:` URL) should pass CSP. `chrome-extension:` is NOT in `img-src`, so strategy 2 (packaged PNG) depends on Chrome's extension-resource CSP bypass for `web_accessible_resources` — it is the riskier one, not the safer one as originally assumed.
  - The full filter chain (feFlood `#808080` → feImage data: map → feMerge → feDisplacementMap, sRGB) was verified in real Chrome via headless screenshot of `test/sandbox.html` — visible lensing, no uniform shift, straight lines when off. See `test/shot-on.png` / `test/shot-off.png`.
  - **RESULT on real x.com (verified live 2026-06-11): all three strategies PASS, zero CSP violations.** lens-data (feImage + canvas `data:` URL): PASS, 688 px displaced in the probe. lens-packaged (feImage + `chrome.runtime.getURL` PNG): PASS. spaghetti (CSS transforms): works. **Winner/default: lens-data** — no packaged asset needed, map can be regenerated as mass changes; spaghetti kept as a selectable `renderMode` and as the auto-degrade fallback; lens-packaged is a known-good spare but not wired as a mode.
- **Viewport anchoring (post-spike fix):** the hole lives in VIEWPORT coordinates (it must agree with the `position: fixed` overlay canvas to the pixel). Every animation frame, inside the rAF loop, convert viewport → filtered-element local coords (`holeViewport − column.getBoundingClientRect().left/top`) and update feImage x/y from that — never from a debounced scroll listener, or the warp lags/swims during fast scrolls. Spaghetti mode needs no conversion: `getBoundingClientRect()` on tweets is already viewport-relative. Original spike bug for the record: `lastX`/`lastY` change-detection was initialized to `NaN`, and `NaN > 0.5` comparisons are always false, so feImage x/y never updated after frame one — the hole stuck in content space and scrolled away.
- **feImage gotcha — uncovered regions:** where the map image doesn't cover the filter region, `in2` reads transparent black (channel 0) → a uniform `-scale/2` shift of the WHOLE element. Fix: composite the map over a neutral `feFlood flood-color="#808080"` via feMerge.
- **feImage gotcha — color space:** default `color-interpolation-filters` is linearRGB, which destroys the 128-neutral encoding. The filter MUST set `color-interpolation-filters="sRGB"`.
- **feImage positioning:** with `primitiveUnits="userSpaceOnUse"`, feImage x/y/width/height are CSS px relative to the filtered element's border-box top-left — per-frame hole motion = two cheap `setAttribute` calls, no canvas regeneration.
- X is a React SPA: virtualized scrolling + client-side navigation. The timeline container may not exist at injection and does not survive navigation. Re-acquire `main[role="main"]` (and primary column within) via MutationObserver + periodic poll.
- Never filter `body` — fixed-position elements break (SVG filter creates a containing block) and perf tanks. Filter the timeline column only.
- X's virtualization can rewrite inline styles on `article` elements — in spaghetti mode, set transforms on a wrapper we control, and restore styles when elements leave the radius or on disable.
- MV3 service worker is ephemeral: persist everything to `chrome.storage.local`; use `chrome.alarms` (not setTimeout/setInterval) for the decay tick.
- **Orphaned content scripts (found the hard way, 2026-06-11):** reloading the extension on chrome://extensions does NOT reload open x.com tabs — the old content script stays alive there (rAF + DOM still work, chrome.* is dead) while Chrome injects the new one alongside. Symptom: a "ghost" hole driven by old code at positions the new motion can't produce, plus a dead warp because two filters share the `event-horizon-lens` id and the visible hole belongs to the other script. Defense in both directions: at startup `evictOrphanRemnants()` (content.ts) removes leftover overlay/panel/SVG nodes and clears a stale column filter; live scripts poll `chrome.runtime?.id` (throws/undefined once orphaned — every 2 s in the rAF tick, plus the reduced-motion poll, spike-panel interval, and heartbeat) and tear themselves down completely. `FilterLensRenderer` also refuses to "save" a column filter value that contains its own filter id. The spike panel + a `console.info` at injection print `__EH_BUILD__` (git SHA + build time, injected by build.mjs) so a stale build is identifiable at a glance.
- Displacement map encoding: deflection vectors in R/G channels around the 128 neutral point; magnitude ∝ mass / max(r, eventRadius)². The vector is radial pull **plus a tangential swirl component** (SWIRL 0.8 — wraps content around the hole like light orbiting the photon sphere; pure inward pull reads as "suction", not lensing); the summed vector is clamped to unit length because the 128±127 encoding can't hold more. Mirrored in tools/gen-displacement.mjs — keep in sync.
- **Chromatic aberration is real, not drawn (2nd reference pass):** the filter splits SourceGraphic into R/G/B via feColorMatrix, displaces each at scale ×1.06/×1.0/×0.94, and recombines with additive feComposite arithmetic — lensed content carries genuine rainbow fringing. Far from the hole the three channels align exactly, so untouched content is pixel-identical. Cost: 3 displacement passes ≈ 3× filter raster; the degrade ladder is the safety net. Tuning: `INFLUENCE_FACTOR` 3.5, `STRENGTH` 0.12, `MAX_SCALE` 200 — the map's saturated core (r < √STRENGTH normalized) intentionally pokes out past the disc (disc = 1/3.5 = 0.286 < 0.346) as a thin max-shift band where content wraps fully around the hole, like the reference's inner rings. Keep disc ≥ ~0.83·√STRENGTH of influence or the band gets fat and looks like a glitch.
- **The map is baked ONCE at reference mass 1 and never regenerated**; live intensity rides the per-frame `scale` attribute as `MAX_SCALE · mass^WARP_GAMMA` (γ=0.4 — front-loaded so the warp is obvious in the first minutes; the old map-mass × scale-mass product gave mass² intensity, ~1% at mass 0.1). Bug record: regenerating the map at mass-tier crossings from the *instantaneous* mass made the warp history-dependent — mass 0.5 approached from above baked a ~25% stronger map than approached from below (user-reported as "1.0→0.5 looks different from 0.5"). A static reference map is immune, and per-frame motion stays attribute-only (feImage x/y), never canvas work. The saturated map core (vector clamp) sits entirely under the black disc (disc = influence/2.6 > saturation radius), so it's never visible.
- Perf: < 4 ms scripting/frame budget; rolling frame-time avg > ~12 ms → auto-degrade lens → spaghetti → overlay-only (log once).
- Pause all work (rAF stops) when tab hidden; respect `prefers-reduced-motion`.
- Modal/composer overlays: suspend the filter while open — lensing a half-typed reply is hostile.

## Key decisions
- `LensRenderer` interface (src/renderer.ts) so render strategies are swappable; **lens-data (feImage + data: URL) is the default** per the Milestone 0 result, spaghettification always available as a setting (`renderMode: "lens" | "spaghetti" | "auto"`). Renderers take a `Hole` in viewport coordinates each frame and own their column acquisition and coordinate conversion.
- Heartbeat model (content → worker every 5 s while visible+focused, scroll delta ≥ 150 px ⇒ 1.5× accrual) instead of tab-time APIs: keeps permissions at `storage` + `alarms` only.
- **Multi-tab dedupe:** the worker credits wall-clock elapsed since the last accrual from ANY tab (capped at one heartbeat interval after a gap), not a fixed 5 s per heartbeat — two visible X tabs sum to real time, never 2×. Handlers are serialized through a promise queue so simultaneous heartbeats can't read the same pre-state.
- **Restart-proof decay:** the mass model is pure functions over persisted state (`src/mass.ts`, unit-tested in `test/mass.test.mjs` via `npm test`). Every worker event loads from `chrome.storage.local`, applies, writes back; decay integrates over real elapsed time since `max(lastDecayAt, lastHeartbeatAt)`, so a killed worker or closed browser catches up the full gap on the next alarm fire (alarms persist) or `onStartup`. `effectiveSeconds` caps at `limitMinutes*60` so a long binge doesn't take hours to starve.
- Mass curve: linear in `effectiveSeconds/limit`, plus an `0.18 × smoothstep(0.7, 1, raw)` push — accelerating endgame, full mass lands slightly before the nominal limit.
- **Clock-skew dt-cap:** a single decay step never integrates more than `MAX_DECAY_STEP_MINUTES` (15), so a laptop waking from hours of sleep doesn't instantly starve the hole; per-minute alarm cadence while awake is far below the cap, so "off X ~30 min → gone" is unaffected. Feeding is capped symmetrically (one heartbeat interval per accrual); backwards clock jumps are no-ops in both directions. All tested.
- **Hole pipeline (M2):** `HoleController` (src/hole-controller.ts) is the single source of the hole: reads mass from `storage.onChanged`, eases it (exp smoothing, τ=700 ms — never pops), computes disc radius (`πR² = maxCoverage·W·H` at mass 1, area linear in mass), drifts via `HoleMotion`, fades in below mass 0.05, draws via `HoleOverlay` (fixed full-viewport canvas, z-index below the spike panel, pointer-events:none). rAF stops when the tab is hidden OR mass ≈ 0 (restarts on storage change); prefers-reduced-motion → no rAF at all, single static render per change. Renderers consume `controller.currentHole()` so disc and distortion agree to the pixel.
- **Debug mass override:** spike panel (`?bhspike` / Ctrl+Shift+B only — ships inert) has a 0..1 slider calling `HoleController.setMassOverride()`, bypassing the tracker for visual tuning. Overlay visuals verified by headless screenshots of `test/overlay-sandbox.html` at mass 0.15/0.5/1.0.
- Headless-Chrome test quirks: `--screenshot` renders black on window-scrolled pages (use the sandbox `?offset=N` margin trick); ES-module imports are CORS-blocked on `file://` (bundle IIFE with `--global-name`).
- **Headless + rAF (M3):** `--virtual-time-budget` is USELESS for the rAF-driven pipeline — rAF timestamps barely advance under virtual time, so `dt`-based easing freezes near 0 while timers race ahead, and the final screenshot composite can be a stale frame that contradicts the page's actual state. `--dump-dom` doesn't advance rAF at all. The working rig (test/integration-sandbox.html): run in REAL time, hold the load event open with `?stall=1` (an `<img>` pointed at TEST-NET-1 `192.0.2.1`, which hangs on connect) so `--timeout=N` decides the capture moment, and read state from the on-screen debug panel + `console.log` via `--enable-logging=stderr`.
- **Render pipeline (M3):** `RenderManager` (src/render-manager.ts) is the single owner of renderer lifecycle. The controller calls `manager.frame(hole)` inside its rAF (right after the overlay draw, same `Hole` — disc and distortion agree by construction) and reports each frame's cost via `recordFrame(scriptMs, rawDt)`. A renderer exists only while the hole is visible (create at mass > 0.02, dispose < 0.01 — hysteresis so a hovering mass doesn't churn filter setup); disposal restores every DOM side effect, so overlay-only/starved/suspended leave x.com untouched.
- **Degrade ladder (M3):** lens → spaghetti → overlay-only. Cost metric = our scripting time **plus rAF-interval overshoot beyond 17.5 ms** — the SVG filter's raster cost never shows up in JS time, only as missed vsync, so dt overshoot is the only available signal for it. Rolling window 48 frames, degrade when avg > 12 ms, 30 warmup frames after every renderer swap (filter/map setup jank isn't steady state), samples with dt > 100 ms discarded (tab restore ≠ perf data). Degrading is unconditional across render modes (the budget is a hard constraint, not a preference); changing `renderMode` in options forgives past degradation; a spike-panel pin disables degrade so tuning sessions aren't yanked away. Each step logs once via `console.info`. Verified end-to-end headlessly with `?slow=30` synthetic jank: lens → spaghetti at avg 16.0 ms, → overlay-only at 16.4 ms, transforms restored, no oscillation.
- **Spaghetti is IO-driven (M3):** an IntersectionObserver (rootMargin 200px) maintains the `visible` set; a MutationObserver on the column feeds newly virtualized-in `article`s to the IO. The per-frame loop touches only the visible set — never walks the column. Restore on: leaving the expanded rect (IO), leaving the influence radius (t ≤ 0), column swap (SPA nav), and dispose; disconnected nodes are dropped (X took our styles with them).
- **Motion & look restyle (post-M3 user feedback, 2026-06-11):** the hole wanders anchored to the FEED COLUMN horizontally (column center ± 0.6·columnWidth — overshoots into the sidebars but always keeps a bite of the feed) and the full viewport height vertically; each axis sums three incommensurate sines (periods ~9–30 s, quasi-random Lissajous wander); the (wx,wy) wander vector is normalized into the unit disc so both axes can never peak together — screen corners are geometrically unreachable; a hard clamp keeps the disc FULLY on screen (`margin = discR + 12`) — after two "it went off-screen" reports, any off-screen lean loses more than it gains. TWO motion mistakes on record: (1) the original two-octave drift (57–217 s periods) sat near a path extreme for a minute — read as "drifted into a corner and died"; drift periods must stay well under a minute. (2) The first restyle roamed the whole viewport — on wide screens the hole spent most of its time over the dark sidebars, where a black disc is invisible and the column-scoped filter has nothing to warp; user-reported as "it moves off screen and the effect disappears". The hole must stay where the content is. Warp exaggeration: `MAX_SCALE` 150 (was 85), `STRENGTH` 0.085 (was 0.05), `INFLUENCE_FACTOR` 2.6 (was 2.2), plus the displacement-map swirl. Overlay matches the Ghostty-reference look: tight accretion glow (surroundings stay dark), 3 chromatic ring echoes (RGB-offset arcs, alpha falling with distance), brighter photon ring, and a white-hot accretion hotspot orbiting the ring at 0.55 rad/s (`tSec` param on `HoleOverlay.draw`; pass a constant under reduced motion → static hotspot).
- **Grok-style orbit ring (user request, 2026-06-11):** a tilted elliptical accretion ring (semi-axes 1.7×/0.5×discR, tilt −0.42 rad — the Grok logo diagonal). Drawn in two halves for occlusion: back half (ellipse param π..2π, the upper arc) BEFORE the black disc and dim; front half AFTER the disc, bright with bloom — Saturn/Gargantua look. The hotspot now travels this ellipse (same parametric angle; front/back by sin θ), dimmer and smaller on the far side. Reduced motion = static phase, as before.
- **appearAfterMinutes (user request):** grace period in the mass model — `mass = curve((effectiveSeconds − appear·60) / (limit·60))`, accrual cap raised to `(appear+limit)·60`, so the hole is invisible until you've fed it `appear` minutes, then grows over `limit` minutes; decay shrinks the same budget so time away re-earns the grace. Default 0 (immediate). Set in options ("Appears after"). Tested (grace gating + cap + re-earn).
- **Modal/composer suspension (M3, pulled forward from M4):** `renderFrame` polls `[aria-modal="true"]` at most every 300 ms; while open, the *displayed* target mass is 0 — the hole eases out smoothly and the manager's mass threshold tears down the filter ≈ 2 s later. Mass tracking is untouched. The rAF loop stays alive during suppression (the idle-stop condition checks the real target, not the suppressed one), which is what notices the modal closing. Reduced-motion has no rAF, so a 600 ms interval polls for modal open/close and triggers a static re-render.
- Options page doubles as the live status display (`hole mass N% · M min on X today` via `storage.onChanged`) — the manual test surface for decay without DevTools.
- Closing the tab is the reset — deliberately no reset button (product decision, see PRODUCT.md).
